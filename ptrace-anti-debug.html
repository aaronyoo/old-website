Ptrace Anti-Debug

1. Ptrace Anti-Debug Technique

The other day I was reversing a binary that used a neat anti-debug trick involving ptrace. Although this simple form of anti-debug is well known in the reversing community, if you are new (like I am) it is quite a cute trick. The “ptrace anti-debug trick”, as I shall call it from now on, allows a program to alter its behavior if it detects that it is being run under a debugger. This makes it a bit more difficult to reverse the executable as you can no longer reliably dynamically analyze it, as it could take an abnormal code path or just exit when it detects the presence of a debugger.

So how does it work? The ptrace(2) system call is normally used by a debugger on linux machines. Specifically, it provides an interface and means for one process to control another process. This is how a debugger is able to control another program. For example, when I have a bug in my code, I fire up a linux debugger which will call ptrace to take control of my program so that I can issue commands from the debugger. The relationship that ptrace creates can be represented by a directed graph. The below figure indicates that the “Debugger” is tracing the “Program”: 

Debugger --ptrace--> Program

For the ptrace anti-debug mechanism, the program tries to call ptrace on itself. This will fail if the program is being debugged and will succeed if the program is not being debugged because no two tracers can trace the same tracee. Or equivalently, no two debuggers can debug the same process. Note that this is not the same as debugging two different instances of the same process. Doing the aforementioned is perfectly fine. Only debugging the exact same running instance of a program from two different debuggers is illegal. The program using this anti-debug trick tries to create the following scenario, which is disallowed: 

Debugger --ptrace--> Program -----
                        ^        |
                        |        |
                        --ptrace--

Essentially, the program can call ptrace to try to trace itself, and if the call fails, that means that it is probably already being traced by a debugger. Here is a simple example using ptrace(2): 

#include <stdio.h>
#include <sys/ptrace.h>

int main() {
    if (ptrace(PTRACE_TRACEME, 0, 1, 0) == -1) {
        printf("Debugger\n");
    } else {
        printf("Normal\n");
    }
    return 0;
}

Here, there are two branches, one for normal execution and one for execution under a debugger. If the above program is run normally, it will output “Normal” but if it is run with a debugger like GDB, it will output “Debugger”.

2. Defeating Ptrace Anti-Debug

So now the question is, how can we get around the ptrace anti-debugging technique? There are two well known ways to circumvent this technique, namely:

  1. Edit the binary (patch or jump PC)
  2. Use a linker preload

2.1. Edit the Binary

One simple way to deal with the ptrace call is to patch it out with a NOP or add an unconditional jump to the “Normal” branch so that normal behavior is observed even under debug. For example, here is the disassembly of the main function for the compiled binary: 

000000000000068a <main>:
 68a:   55                      push   rbp
 68b:   48 89 e5                mov    rbp,rsp
 68e:   b9 00 00 00 00          mov    ecx,0x0
 693:   ba 01 00 00 00          mov    edx,0x1
 698:   be 00 00 00 00          mov    esi,0x0
 69d:   bf 00 00 00 00          mov    edi,0x0
 6a2:   b8 00 00 00 00          mov    eax,0x0
 6a7:   e8 b4 fe ff ff          call   560 <ptrace@plt>
 6ac:   48 83 f8 ff             cmp    rax,0xffffffffffffffff
 6b0:   75 0e                   jne    6c0 <main+0x36>

The way that the compiler generated the assembly, we just have to replace the call to ptrace with NOPs to get around the anti-debug. Another possbile approach is to move the program counter when under debug to jump to the normal branch before the call to ptrace happens. Moving the program counter is very similar in spirit to patching, but is also dangerous for the consistent execution of the program. In general, both of these approaches are good for singular or few uses of ptrace. However, these approaches do not work well when ptrace is called in multiple places.

2.2. Using a Linker Preload

A linker preload is a way to instruct the linker to link certain specified shared libraries before all others (even before the C runtime itself). In general, this gives very granular control over symbol resolution and can be used to wrap function calls, implement alternative versions of library functions (often done with malloc), or technically even re-define functions without re-compiling (not advised). Let's see how we can use it to define our own ptrace function and have the binary call that one instead of the one defined by the C standard libaries. Here is our definition of ptrace that simply returns 0: 

// This is a redefinition of ptrace to get around the
// ptrace anti-debug trick.

long ptrace(int a, int b, void *c, void *d) {
        return 0;
}

Now we just need to compile our C file into a shared library: 

$ gcc -shared -fPIC -o ptrace_inject.so ptrace_inject.c

Now if we run the program under debug we get the “Normal” output because the call to ptrace resolves to our definition of ptrace which always returns 0: 

(gdb) file a.out
Reading symbols from a.out...(no debugging symbols found)...done.
(gdb) set environment LD_PRELOAD ./ptrace_inject.so
(gdb) r
Starting program: /home/aaron/Desktop/a.out 
Normal
[Inferior 1 (process 3484) exited normally]
(gdb)

And for reference, here is the canonical command to run from the command line with a linker preload: 

$ LD_PRELOAD=$PWD/ptrace_inject.so ./a.out

Linker preloading is extremely powerful, and in this case, it can help change symbol resolution to get around the ptrace anti-debug technique. Also, notice that we didn't have to recompile the binary in order for this to work since it relies on the linker.
